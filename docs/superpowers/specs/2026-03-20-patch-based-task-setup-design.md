# Patch-Based Task Setup

**Date:** 2026-03-20
**Status:** Draft

## Problem

Task `setup.sh` files are hard to read because they embed file content inline using shell heredocs (`cat > file.sql << 'EOF'...EOF`) or scatter file modifications across multiple `sed` lines. The reader must mentally execute shell code to understand what state the agent will encounter at the start of a task.

The goal is to make the *changes a task makes to the shared project* legible at a glance.

## Approach

Introduce a convention: tasks may store a `setup/changes.patch` file (unified diff format) that `setup.sh` is responsible for applying. The patch encodes file changes declaratively as a diff against the shared project baseline — readable, diffable in git, and generated by standard tooling.

This is a **pure convention change**. No harness modifications required. `setup.sh` remains in control and calls `patch` explicitly.

## File Layout

```
tasks/airbnb003/
├── task.yaml
├── setup.sh              ← applies patch + runs commands
└── setup/
    └── changes.patch     ← unified diff against shared project baseline
```

The `setup/` directory is already copied into the container before `setup.sh` runs, and is deleted by the harness afterwards. No new lifecycle handling needed.

## Patch Format

Standard unified diff, generated with `git diff` or `diff -u`. Paths are relative to the dbt project root (the working directory when `setup.sh` runs).

Example patch (`setup/changes.patch`):
```diff
--- a/models/source/src_hosts.sql
+++ b/models/source/src_hosts.sql
@@ -1,8 +1,0 @@
-{{
-	config(
-		materialized="table" ,
-		alias="src_hosts" ,
-		schema="main" ,
-		unique_key="HOST_ID"
-	)
-}}
-
```

## Applying the Patch in setup.sh

`setup.sh` calls `patch -p1` to apply the diff:

```bash
#!/bin/bash
patch -p1 < /app/setup/changes.patch
dbt deps
dbt run
```

The `-p1` flag strips the `a/`/`b/` path prefix produced by `git diff`. The `setup/` directory is available at `/app/setup/` during `setup.sh` execution.

## Generating Patches

To generate a patch for a task, diff a modified copy of the shared project against the original:

For the common case of overwriting files wholesale, the simplest workflow is:
1. Copy the shared project to a temp directory
2. Make the desired edits
3. `diff -ru original/ modified/ > tasks/mytask/setup/changes.patch`

## Migration of Existing Tasks

Tasks with `cat > file.sql << 'EOF'` heredocs in `setup.sh` are the primary candidates. The conversion is:

1. Extract each file content from the heredoc into its actual target path
2. Diff against the shared project baseline to produce the patch
3. Replace the heredoc block in `setup.sh` with `patch -p1 < /app/setup/changes.patch`

Setup steps that only use `rm` **do not need migration**

Tasks with complex procedural setup (airbnb010, airbnb011 — watchdog processes, database mutations) should not use patches; `setup.sh` remains the right tool there.

## Scope

This convention applies to the `setup/` phase. It is not extended to solution scripts or migration steps at this time.

## What This Does Not Change

- Harness setup lifecycle (no changes to `trial_handler.py` or `setup_orchestrator.py`)
- `setup.sh` structure or cleanup behavior
- `task.yaml` schema
- Solution scripts (`solution.sh`)
- Pre-test scripts (`test_setup` in `task.yaml`)
